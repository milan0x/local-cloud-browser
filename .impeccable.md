## Design Context

### Users
Individual developers and small DevOps/platform teams who manage local AWS-compatible services (LocalStack, MinIO) during development. They use the app as a daily companion while building and debugging — it needs to stay out of the way and just work. Users range from developers testing locally to teams managing shared local infrastructure.

### Brand Personality
**Professional, clear, reliable.**

The app should feel like a well-crafted native macOS developer tool — quiet confidence, zero theatrics. It prioritizes clarity of information, predictable behavior, and consistency above all. Every interaction should reinforce trust: clear error messages, obvious navigation, and no crashes or surprises. It's friendly without being playful, capable without being showy.

### Emotional Goals
- **Confidence & trust**: "I know exactly what this will do" — predictable, safe, no surprises
- **Calm efficiency**: "This just works" — frictionless, fast, minimal cognitive overhead

### Aesthetic Direction
- **Visual tone**: Native macOS, utilitarian, polished but restrained
- **References**: Proxyman, TablePlus, Xcode, Instruments — clean native macOS dev tools that feel like they belong on the platform
- **Theme**: System appearance (light + dark mode via semantic SwiftUI colors)
- **Typography**: System fonts only (SF Pro, SF Mono) — no custom typefaces
- **Iconography**: SF Symbols exclusively — consistent with macOS platform conventions

### Anti-References (What This Must NOT Look Like)
- **Electron apps**: No web-app-in-a-wrapper aesthetic (Slack, Discord) — must feel genuinely native
- **AWS Console**: No dense, overwhelming information dumps — progressive disclosure instead
- **Generic SaaS dashboards**: No startup admin panel with card grids, gradients, or decorative UI

### Design Principles

1. **Clarity over cleverness**: Every label, error message, and navigation path should be immediately understandable. No fancy UI/UX tricks — the interface should communicate, not decorate.

2. **Native and invisible**: Follow macOS Human Interface Guidelines faithfully. The app should feel like it ships with the OS. Use semantic colors, standard controls, and platform conventions so developers never have to learn a custom design language.

3. **Safety by default**: Read-only mode on, endpoint validation, confirmation dialogs for destructive actions. The UI should make it hard to do something dangerous accidentally. Visual hierarchy should surface warnings and safety state prominently.

4. **Consistency across modules**: All 28+ service modules should feel identical in structure and interaction patterns. Shared components (ListHeaderBar, SearchBarView, StatusBadge, CreateFormScaffold) enforce this — never deviate with one-off patterns.

5. **Information density, not overload**: Show what developers need without drowning them. Use progressive disclosure (disclosure groups, detail panes, tabs) to layer information. Monospaced text for IDs and values, readable fonts for labels and descriptions.

### Accessibility
- macOS system defaults: VoiceOver support, standard keyboard navigation, system font scaling
- Semantic colors for automatic light/dark mode adaptation
- Accessibility labels on all interactive elements (already implemented)
